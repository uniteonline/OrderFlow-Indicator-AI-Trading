# 指标升级方案

## 1. 目标

本方案只针对 `orderflow-indicator-engine` 的原始产出层，也就是输出到 `temp_indicator` 的指标 JSON 结构。

本次升级目标：

1. 保留当前分钟级事件触发语义。
2. 为事件型指标补充最近 `7d` 的全量事件明细。
3. 新增 `fvg` 结构指标，为后续 `llm` 层提供可直接使用的 HTF 低接受价格区。
4. 为后续 `llm` 层做 `7d` 摘要、近端聚合、事件序列分析与结构区过滤提供足够原始上下文。
5. 不在本次方案中处理 `temp_model_input` 的聚合压缩方式。
6. 尽量保持向后兼容，不打断当前依赖旧字段的链路。

## 2. 适用范围

本方案覆盖以下 5 个事件型指标：

1. `divergence`
2. `absorption`
3. `initiation`
4. `buying_exhaustion`
5. `selling_exhaustion`

以及 1 个非事件型结构指标：

1. `fvg`

相关过滤视图指标：

1. `bullish_absorption`
2. `bearish_absorption`
3. `bullish_initiation`
4. `bearish_initiation`

本次不作为主升级对象。它们后续可以基于 `absorption` / `initiation` 的 `recent_7d.events` 做方向过滤得到，不需要重复维护另一套完整 7 天事件仓。

`fvg` 的定位单独说明：

1. `fvg` 只负责输出 HTF 结构区，不作为独立开仓信号。
2. 它最适合作为现有 `imbalance / divergence / absorption / avwap / value area / lvn` 的结构过滤器。
3. 基础版只使用合约市场 HTF K 线判定，不要求 spot 参与基础定义。

## 3. 当前问题

当前原始产出更偏向“当前分钟是否新确认事件”的触发器语义，例如：

```json
{
  "divergence": {
    "signal": false,
    "reason": "no_candidate",
    "candidates": [],
    "latest": null
  },
  "absorption": {
    "event_count": 0,
    "events": [],
    "latest": null
  },
  "initiation": {
    "event_count": 0,
    "events": []
  },
  "buying_exhaustion": {
    "event_count": 0,
    "events": []
  },
  "selling_exhaustion": {
    "event_count": 0,
    "events": []
  }
}
```

这个结构的问题不是“算法不对”，而是“上下文不够”：

1. 只能看当前分钟是否有新事件。
2. 无法直接回答最近 `30m / 4h / 1d / 7d` 发生过什么。
3. 无法在 `llm` 层方便地做连续吸收、吸收失败、发起接力、衰竭反转等序列判断。
4. 一旦当前分钟没有新事件，`event_count=0` 容易让上层误以为“近期没有发生事件”。

## 4. 设计原则

### 4.1 保留旧字段

旧字段全部保留，继续表示“当前分钟刚确认的事件”。

### 4.2 新增 7 天全量事件层

统一新增：

1. `recent_7d.event_count`
2. `recent_7d.events`
3. `latest_7d`

### 4.3 排序统一

`recent_7d.events` 统一按 `event_available_ts` 升序排列：

- 旧 -> 新

这样与 `orderflow-indicator-engine` 现有多数历史序列风格一致，也更适合后续做窗口聚合。

### 4.4 裁切统一

只保留：

- `event_available_ts >= ts_bucket - 7d`

### 4.5 主时间统一

每条事件统一保留以下时间字段：

1. `event_start_ts`
2. `event_end_ts`
3. `event_available_ts`
4. `confirm_ts`

其中：

- 排序和 7 天裁切统一使用 `event_available_ts`

原因：

1. `event_start_ts` 表示事件开始发生。
2. `confirm_ts` / `event_available_ts` 表示事件何时真正可用于交易判断。
3. 后续 `llm` 聚合时应以“何时可交易”为主，而不是“何时开始酝酿”为主。

### 4.6 历史补足策略统一

7 天事件明细的理想来源是完整的滚动历史，但实现上不能假设内存历史永远完整。

统一策略：

1. 第一优先级：使用内存滚动历史
2. 第二优先级：若内存不足，则回补数据库中已落地的 canonical 历史
3. 第三优先级：若内存与数据库都不足，则按当前可见历史输出部分结果

约束：

1. 禁止因为历史不足而直接返回空结果
2. 禁止把部分结果伪装成完整 7 天结果
3. 必须显式输出覆盖度字段，供后续 `llm` 聚合判断可信度

### 4.7 原始层不做摘要替代明细

`temp_indicator` 层本次只负责提供：

1. 当前分钟触发结果
2. 最近 `7d` 全量事件明细
3. 最近一条 `latest_7d`

不在原始层把 7 天事件先压成摘要，以免后续 `llm` 层失去聚合自由度。

### 4.8 原始层优先保留 raw 值，同时允许补充 normalized 值

对于会被后续 `llm` 聚合、人工复盘、策略调优重复使用的字段，原始层必须优先保留 raw 值。

典型场景：

1. `spot_whale_push`
2. `spot_cvd_push`
3. `delta_drop / delta_lift`
4. `break_mag_ticks`
5. `price_push_ticks`

原则：

1. 原始层优先保留 raw 数值
2. 如果已有 normalized / confirm score，也可以同时保留
3. 禁止只保留 normalized score 而丢掉 raw 值

这样后续 `llm` 层仍然可以自由地：

1. 自己做标准化
2. 自己做 30m / 4h / 1d / 7d 聚合
3. 区分“同样 confirm=1.0 但 raw notional 完全不同”的事件强弱

### 4.9 7 天事件明细的数据来源架构

本次方案明确锁定：`recent_7d.events` 不依赖数据库，不从 `feat.indicator_snapshot` 反查，也不从 event 表回拼。

统一数据来源架构：

1. 在 `indicator_engine` 内部直接基于当前滚动历史重新运行对应检测函数
2. 为各事件型指标新增 `_all_history()` 或等价变体
3. 该变体移除“必须等于当前分钟”的过滤，只返回当前可见历史中全部已确认事件
4. 最终在 snapshot 组装阶段按：
   - `event_available_ts >= ts_bucket - 7d`
   裁切成 `recent_7d.events`

这样做的原因：

1. 与当前分钟检测函数保持同源、同口径
2. 不引入 DB 依赖，不会出现回放/持久化与实时计算两套逻辑
3. 后续 replay/backfill 时，7 天事件明细可随历史一起重建
4. 便于做统一去重与排序

统一要求：

1. `current` 语义：
   - 继续使用“当前分钟新确认事件”
2. `recent_7d` 语义：
   - 使用 `_all_history()` 检测结果裁切后的最近 7 天已确认事件

不能把当前分钟 `events` 直接累积成 7 天数组，也不能依赖外部存储去拼装。

此外必须补充覆盖度字段：

1. `lookback_requested_minutes`
2. `lookback_covered_minutes`
3. `lookback_missing_minutes`
4. `lookback_coverage_ratio`
5. `history_source`

其中 `history_source` 建议枚举：

1. `memory_only`
2. `memory_plus_db`
3. `partial_memory_only`
4. `partial_memory_plus_db`

### 4.10 FVG 的市场边界与判定边界

`fvg` 不属于“最近 7 天事件仓”这一类指标，而是持续存在并随时间更新状态的结构区指标。

本次明确锁定：

1. 主计算市场：
   - 合约 `futures`
2. 基础版不需要：
   - spot kline
   - spot confirm
3. 必需输入：
   - 合约 HTF OHLCV：`1h / 4h / 1d`
   - ATR 所需 OHLC
4. 可选增强：
   - spot kline
   - AVWAP
   - LVN
   - Value Area
   这些只用于下游共振打分，不参与基础 FVG 判定。
5. FVG 的基础定义是“位移后留下的低接受价格区”，不是独立交易信号。
6. FVG 的基础状态更新按同一 HTF 的已收盘 bar 序列执行；基础版不使用未收盘 HTF bar 提前确认。

## 5. 统一输出骨架

### 5.1 通用事件型指标骨架

适用于：

1. `absorption`
2. `initiation`
3. `buying_exhaustion`
4. `selling_exhaustion`

建议结构：

```json
{
  "event_count": 0,
  "events": [],
  "latest": null,
  "recent_7d": {
    "event_count": 0,
    "events": []
  },
  "latest_7d": null
}
```

语义：

1. 顶层 `event_count/events/latest`：当前分钟新确认事件
2. `recent_7d.events`：最近 `7d` 全量事件明细
3. `latest_7d`：最近 `7d` 中最新一条事件

### 5.2 divergence 特殊骨架

`divergence` 目前已有 `signal/reason/candidates/latest`，建议扩成：

```json
{
  "signal": false,
  "reason": "no_candidate",
  "candidates": [],
  "latest": null,
  "recent_7d": {
    "event_count": 0,
    "events": []
  },
  "latest_7d": null
}
```

语义：

1. 顶层旧字段：继续表示当前分钟是否出现新可用 divergence
2. `recent_7d.events`：最近 `7d` 已确认 divergence 事件明细
3. `latest_7d`：最近 `7d` 最新一条 divergence

### 5.3 fvg 结构骨架

`fvg` 建议输出为多窗口结构区视图：

```json
{
  "source_market": "futures",
  "base_detection_uses_spot": false,
  "by_window": {
    "1h": {
      "fvgs": [],
      "active_bull_fvgs": [],
      "active_bear_fvgs": [],
      "nearest_bull_fvg": null,
      "nearest_bear_fvg": null,
      "requested_bars": 64,
      "returned_bars": 64,
      "missing_bars": 0,
      "coverage_ratio": 1.0,
      "history_source": "memory+db",
      "min_required_bars": 16,
      "is_ready": true
    },
    "4h": {
      "fvgs": [],
      "active_bull_fvgs": [],
      "active_bear_fvgs": [],
      "nearest_bull_fvg": null,
      "nearest_bear_fvg": null,
      "requested_bars": 64,
      "returned_bars": 64,
      "missing_bars": 0,
      "coverage_ratio": 1.0,
      "history_source": "memory+db",
      "min_required_bars": 16,
      "is_ready": true
    },
    "1d": {
      "fvgs": [],
      "active_bull_fvgs": [],
      "active_bear_fvgs": [],
      "nearest_bull_fvg": null,
      "nearest_bear_fvg": null,
      "requested_bars": 64,
      "returned_bars": 64,
      "missing_bars": 0,
      "coverage_ratio": 1.0,
      "history_source": "memory+db",
      "min_required_bars": 16,
      "is_ready": true
    }
  }
}
```

语义：

1. `1h / 4h / 1d`
   - `4h`：主交易结构
   - `1d`：大方向 / 大级别目标区
   - `1h`：辅助精细定位
2. 每个窗口的主输出必须先是数组：
   - `fvgs[]`
   该数组列出该时间窗口内当前可参考的全部 FVG，每条用 `side` 区分 `bull/bear`。
3. `active_*_fvgs`
   - 只保留当前仍有效的结构区，状态限定在：
     - `fresh`
     - `partially_filled`
     - `mitigated`
   - 它们是 `fvgs[]` 的按方向过滤视图，不应是另一套独立数据源。
4. `nearest_*_fvg`
   - 从对应 `fvgs[]` 过滤后的同侧 active 集合中按当前价格到区间的最小距离选出
5. 每条 FVG 建议保留稳定主键：
   - `fvg_id`
   便于跨分钟去重、回放核对和状态追踪
6. 每个窗口还必须带 readiness 元数据：
   - `requested_bars / returned_bars / missing_bars / coverage_ratio / history_source`
   - `min_required_bars`
   - `is_ready`
7. `is_ready` 固定定义为：
   - `returned_bars >= min_required_bars`
   - 若 `is_ready = false`，该窗口仍保留 coverage 元数据，但 `fvgs[] / active_*_fvgs` 必须为空，`nearest_*_fvg = null`

### 6.2.4 absorption 专项说明

`absorption` 的 `spot_whale_push` 本次明确采用“双轨保留”：

1. 保留 raw：
   - `spot_whale_push_notional`
2. 同时保留归一化确认分数：
   - `spot_whale_confirm_score`

原因：

1. raw 值适合后续多窗口聚合、横向比较、人工复盘
2. normalized 值适合 `llm` 后续快速理解 confirm 强弱
3. `spot_confirm` 是布尔结论位，建议固定定义为：
   - `spot_flow_confirm_score > 0.0 || spot_whale_confirm_score > 0.0`

同时，原方案中的单字段：

- `stacked_imbalance_side`

本次升级为更完整的触发时 stacked 状态：

1. `stacked_buy_trigger`
2. `stacked_sell_trigger`
3. `max_buy_stack_len_trigger`
4. `max_sell_stack_len_trigger`

这样后续 `llm` 层可以区分：

1. 只是方向标签
2. 还是带有明显 stacked 强度的吸收事件

### 6.6.4 基础判定公式

对任意时间框架 `w in {1h, 4h, 1d}`，输入 bar 序列为：

1. `O_w[t]`
2. `H_w[t]`
3. `L_w[t]`
4. `C_w[t]`
5. `Vol_w[t]`

基础参数默认值：

1. `epsilon_gap_ticks = 2`
2. `atr_lookback = 14`
3. `min_body_ratio = 0.60`
4. `min_impulse_atr_ratio = 1.30`
5. `min_gap_atr_ratio = 0.15`
6. `max_gap_atr_ratio = 1.20`
7. `mitigated_fill_threshold = 0.80`
8. `invalid_close_bars = 1`
9. `eps = 1e-12`
10. `min_required_bars = atr_lookback + 2`
    - 默认 `atr_lookback = 14` 时，最小技术门槛为 `16`

基础定义：

1. `TR[t] = max(H[t]-L[t], abs(H[t]-C[t-1]), abs(L[t]-C[t-1]))`
2. `ATR[t] = mean(TR[t-j], j=0..atr_lookback-1)`
3. `epsilon_gap = epsilon_gap_ticks * tick_size`
4. Bull FVG 条件：
   - `L[t] > H[t-2] + epsilon_gap`
5. Bear FVG 条件：
   - `H[t] < L[t-2] - epsilon_gap`
6. `body_ratio_mid = abs(C[t-1]-O[t-1]) / (H[t-1]-L[t-1] + eps)`
7. 多头 `close_location_mid = (C[t-1]-L[t-1]) / (H[t-1]-L[t-1] + eps)`
8. 空头 `close_location_mid = (H[t-1]-C[t-1]) / (H[t-1]-L[t-1] + eps)`
9. `impulse_atr_ratio = TR[t-1] / (ATR[t-2] + eps)`
10. `width_atr_ratio = width / (ATR[t-1] + eps)`

基础过滤条件：

1. `body_ratio_mid >= min_body_ratio`
2. `close_location_mid >= 0.70`
3. `impulse_atr_ratio >= min_impulse_atr_ratio`
4. `min_gap_atr_ratio <= width_atr_ratio <= max_gap_atr_ratio`

边界定义：

1. Bull FVG：
   - `lower = H[t-2]`
   - `upper = L[t]`
2. Bear FVG：
   - `lower = H[t]`
   - `upper = L[t-2]`
3. `mid = (lower + upper) / 2`
4. `width = upper - lower`

位移强度分数：

1. `displacement_score = clip01(0.4 * body_ratio_mid + 0.3 * min(1, impulse_atr_ratio / 2.0) + 0.3 * close_location_mid)`

### 6.6.5 fill_pct、state、失效与最近区

回补比例：

1. Bull FVG：
   - `fill_pct = clip01((upper - min_low_since_birth) / (width + eps))`
2. Bear FVG：
   - `fill_pct = clip01((max_high_since_birth - lower) / (width + eps))`

状态定义：

1. `fill_pct == 0`
   - `fresh`
2. `0 < fill_pct < mitigated_fill_threshold`
   - `partially_filled`
3. `mitigated_fill_threshold <= fill_pct < 1`
   - `mitigated`
4. `fill_pct >= 1`
   - `fully_filled`
5. 若后续出现有效收盘穿越对侧边界：
   - `invalidated`

失效条件：

1. Bull FVG：
   - 连续 `invalid_close_bars` 根收盘低于 `lower - epsilon_gap`
2. Bear FVG：
   - 连续 `invalid_close_bars` 根收盘高于 `upper + epsilon_gap`

年龄与触碰：

1. `age_bars = current_closed_bar_index - birth_index`
2. Bull 触碰条件：
   - `L[u] <= upper`
3. Bear 触碰条件：
   - `H[u] >= lower`
4. `touch_count` 为出生后满足触碰条件的 bar 数量累计
5. `fill_pct / touch_count / state / age_bars` 每分钟都从当前可得的已收盘 HTF bar 序列无状态推导，不要求在 `StateStore` 中持久维护跨分钟 FVG 状态

最近区选择：

1. `nearest_bull_fvg`
   - 从 `fvgs[]` 中过滤 `side=bull` 且仍 active 的集合后，按当前价格到区间 `[lower, upper]` 的最小距离选出
2. `nearest_bear_fvg`
   - 从 `fvgs[]` 中过滤 `side=bear` 且仍 active 的集合后，按当前价格到区间 `[lower, upper]` 的最小距离选出

### 6.6.6 FVG 专项说明

FVG 的基础版必须坚持以下边界：

1. 基础判定只使用合约 HTF K 线，不引入 spot 数据。
2. `spot kline / avwap / lvn / value area` 只作为可选增强，不参与基础 FVG 是否成立的判断。
3. FVG 不是开仓信号，而是结构区过滤器。
4. `active_*_fvgs` 只保留当前仍可交易参考的结构区：
   - `fresh`
   - `partially_filled`
   - `mitigated`
5. 每个窗口的第一输出形态必须是数组：
   - `fvgs[]`
   不能只给一条 `nearest_*` 或单个 bull/bear object。
6. `fully_filled` 与 `invalidated` 不进入 active 数组，但仍可在原始审计数据中保留其最后状态。
7. FVG 窗口只有在 `is_ready = true` 时才可被上层视为“可交易结构区输入”。

## 11. 编码前必须锁定的约束

这是本次方案在真正进入代码修改前必须固定下来的实现约束。缺少这些约束，后续很容易出现重复事件、口径漂移或 `llm` 聚合重复计数。

### 10.1 recent_7d 只收“已确认并可交易”的事件

`recent_7d.events` 只能包含：

1. 已完成确认
2. 已具备 `event_available_ts`
3. 已可用于交易判断

明确禁止放入：

1. `divergence.candidates`
2. 尚未 confirm 的半成品事件
3. 仅用于中间计算的临时候选

### 10.2 event_count 必须可直接校验

统一约束：

```text
recent_7d.event_count == recent_7d.events.len()
```

不允许：

1. `event_count` 是另一套统计口径
2. `events` 被裁切但 `event_count` 还是原始总量

### 10.3 latest_7d 必须与 recent_7d 保持一致

统一约束：

1. `recent_7d.events` 升序（旧 -> 新）
2. `latest_7d == recent_7d.events.last()`
3. 若 `recent_7d.events` 为空，则 `latest_7d = null`

### 10.4 必须定义事件唯一键，防止 replay/backfill 重复

由于 `indicator_engine` 存在回放、重算、修复、重启恢复场景，7 天数组如果没有唯一键，后续很容易重复计数。

每条事件都应新增：

```json
{
  "event_id": "..."
}
```

建议规则：

1. `divergence`
   - `event_id = type + pivot_ts_1 + pivot_ts_2 + event_available_ts`
2. `absorption`
   - `event_id = type + event_start_ts + event_end_ts + event_available_ts`
3. `initiation`
   - `event_id = type + event_start_ts + event_end_ts + event_available_ts`
4. `buying_exhaustion / selling_exhaustion`
   - `event_id = type + pivot_ts_1 + pivot_ts_2 + event_available_ts`

要求：

1. 同一事件在 replay/backfill 后 `event_id` 不变
2. `recent_7d.events` 以 `event_id` 去重

### 10.5 当前分钟字段与 7 天字段语义必须切开

必须明确：

1. 顶层旧字段：
   - 当前分钟新确认事件
2. `recent_7d.*`：
   - 最近 7 天所有已确认事件

不能混成：

1. 顶层 `events` 放 7 天明细
2. `recent_7d` 再放另一套不同口径

### 10.6 divergence 的 current 与 recent_7d 口径必须不同

`divergence` 是这 5 个指标里最容易混乱的一个。

统一约束：

1. `candidates`
   - 只保留当前分钟候选
2. `latest`
   - 只表示当前分钟最新可用 divergence
3. `recent_7d.events`
   - 只存最近 7 天已确认 divergence 事件

不能把 `candidates` 原样塞进 `recent_7d.events`。

### 10.7 过滤视图指标不单独维护第二套 7 天仓

这次方案不遗漏：

1. `bullish_absorption`
2. `bearish_absorption`
3. `bullish_initiation`
4. `bearish_initiation`

但它们的 7 天上下文不单独存第二份。

统一约束：

1. 方向过滤视图仍可继续输出当前分钟结果
2. 7 天明细以后由：
   - `absorption.recent_7d.events`
   - `initiation.recent_7d.events`
   通过 `type/direction` 过滤得到

这样不是遗漏，而是避免同一事件维护两份 7 天明细。

### 10.8 事件字段必须尽量直接复用现有 payload

为了避免实现时出现两套不同口径，7 天事件明细建议优先直接复用现有事件 payload 字段，不重新发明一套新数值。

原则：

1. 已有 payload 字段就直接沿用
2. 只新增：
   - `recent_7d`
   - `latest_7d`
   - `event_id`
3. 非必要不改原有单事件字段命名

### 10.9 实施前锁定补充（本轮复核结论）

以下细节在编码前必须锁定，不再作为可选讨论项：

1. `recent_7d.events` 的数据来源
   - 统一来自 `indicator_engine` 内部 re-run 检测函数的全历史结果
   - 为每个目标指标新增 `_all_history()` 或等价变体
   - 去掉“仅当前分钟”过滤
   - 最终按 `event_available_ts >= ts_bucket - 7d` 裁切
   - 不依赖数据库，不从 `feat.indicator_snapshot` 或事件表回拼

2. `absorption.spot_whale_push`
   - 原始层必须同时保留：
     - `spot_whale_push_notional`
     - `spot_whale_confirm_score`
   - 不允许只保留归一化分数

3. `absorption` 的 stacked 触发信息
   - 原始层必须输出：
     - `stacked_buy_trigger`
     - `stacked_sell_trigger`
     - `max_buy_stack_len_trigger`
     - `max_sell_stack_len_trigger`
   - 不再把 `stacked_imbalance_side` 作为唯一表达

4. `initiation` 的 follow-through 信息
   - 原始层必须输出：
     - `follow_through_minutes`
     - `follow_through_delta_sum`
     - `follow_through_hold_ok`
     - `follow_through_max_adverse_excursion_ticks`
   - 这些字段用于表达“突破后是否真的延续”，不能省略

5. 字段命名映射按本方案第 11 节执行
   - 例如：
     - `buying_exhaustion.reject_ratio <- reject_top`
     - `selling_exhaustion.reject_ratio <- reject_bottom`
     - `score <- strength_score_xmk`
   - 编码时不得另起一套同义命名

6. `fvg` 的基础判定只允许使用 futures 已收盘 HTF K 线
   - `1h`
   - `4h`
   - `1d`
   不允许把 spot 或未收盘 HTF bar 混入基础判定

7. `fvg` 的 `nearest_bull_fvg / nearest_bear_fvg` 必须从 active 数组中按价格到区间的最小距离选出
   - 不允许只取最新出生的一条
   - 不允许按数组顺序直接取第一条

8. `fvg` 必须保留稳定的 `fvg_id`
   - 同一结构区跨分钟滚动时 `fvg_id` 不变
   - 只允许状态字段变化
   - 不允许每分钟重生成不同 ID
9. `fvg` 在 `1h / 4h / 1d` 每个窗口下都必须先输出数组：
   - `fvgs[]`
   通过数组列出该窗口内所有 FVG；不允许只保留单条对象

### 11.9 event_id 即使采用 re-run 方案也必须保留

即使 `recent_7d.events` 来自 `_all_history()` 重跑结果，`event_id` 仍然必须保留。

原因：

1. `llm` 层后续做跨分钟聚合时需要稳定去重
2. replay/backfill / 启动回放后，同一事件必须能被识别为同一条
3. 没有 `event_id`，后续 7 天窗口滚动时容易重复统计“上一分钟已经见过的同一事件”

因此：

1. `event_id` 不是可选项
2. 继续按本方案前面定义的规则生成稳定字符串键

## 12. 非目标

本方案暂不处理：

1. `temp_model_input` 的压缩/聚合方式
2. 7 天摘要统计字段，例如：
   - `count_30m`
   - `count_4h`
   - `count_1d`
   - `count_7d`
   - `sequence_state`
3. `bullish_absorption` / `bearish_absorption` / `bullish_initiation` / `bearish_initiation` 的 7 天独立全量仓
4. `fvg` 的 spot/AVWAP/LVN/Value Area 增强评分

这些都应在下一阶段，由 `llm` 层过滤聚合来完成。

## 13. 最终确认

这套原始层升级方案的作用是：

1. 保留当前分钟触发语义
2. 补足最近 `7d` 的全量事件上下文
3. 为后续 `llm` 层做高质量事件聚合提供完整原料

只要严格按本方案保留：

1. 时间字段
2. 方向字段
3. 强度字段
4. 差异值字段
5. spot/futures 归因字段
6. FVG 结构区字段

那么后续 `llm` 层就可以稳定构造：

1. 最近 `30m / 4h / 1d / 7d` 的事件统计
2. 连续吸收 / 吸收失败 / initiation 接力
3. exhaustion 后是否反转
4. divergence 的方向偏置与驱动归因
5. 价格是否正处于 HTF FVG 的回补/失效/再定价区间

这正是中大型交易机会所需要的事件上下文基础。
