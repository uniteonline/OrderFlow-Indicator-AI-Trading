# 指标升级方案代码改造清单

## 1. 目标

本清单用于把 [`指标升级方案.md`](/data/docs/指标升级方案.md) 推进到可直接实施的代码改造层。

本次改造的硬目标：

1. 不妥协原需求。
2. `orderflow-indicator-engine -> temp_indicator` 原始产出层必须输出 5 个主事件指标的最近 `7d` 全量事件明细。
3. 原始层必须新增 `fvg` 结构指标，并输出可直接用于交易过滤的 `1h / 4h / 1d` 合约 HTF 结构区。
4. 原始层必须保留当前分钟语义，同时补足最近 `7d` 上下文与 HTF 结构区上下文。
5. 后续 `llm` 层做摘要、聚合、事件序列判断时，不能因为原始层缺字段而失真。
6. 为了满足需求，可以接受新增字段、恢复事件落库、重放后重建历史；不能为了省存储把需求做弱。

本清单只针对 `indicator_engine` 原始输出层与其直接相关的存储层、发布层、测试层。`llm` 聚合层不在本清单内。

## 2. 代码与数据库现状复核结论

### 2.1 原始产出链路

当前 [`temp_indicator`](#/) 不是直接从数据库生成，而是来自 `indicator_engine` 的 minute bundle MQ 发布。

关键路径：

1. `Dispatcher::process_window(...)` 组装 `IndicatorSnapshotRow`
2. `IndPublisher::publish_minute_bundle(...)` 将 `snapshots[].payload_json` 发布到 MQ
3. `llm` 收到 MQ 后原样写入 `temp_indicator`

涉及文件：

1. [`/data/systems/indicator_engine/src/runtime/dispatcher.rs`](/data/systems/indicator_engine/src/runtime/dispatcher.rs)
2. [`/data/systems/indicator_engine/src/publish/ind_publisher.rs`](/data/systems/indicator_engine/src/publish/ind_publisher.rs)

结论：

- 只要 `IndicatorSnapshotRow.payload_json` 被扩展，`temp_indicator` 就会自动带上新的 `recent_7d` 事件明细。
- 本次不需要在 `llm` 侧为原始层增加额外拼装逻辑。

补充确认：

1. `price_volume_structure` 当前原始产出已经包含：
   - `15m`
   - `4h`
   - `1d`
2. 当前不包含 `3d`
3. 若目标是更好支撑中大型交易机会判断，`pvs` 在原始层应补充 `3d` 窗口
4. 因此本次代码改造清单除 5 个事件指标外，还需纳入 `price_volume_structure.by_window["3d"]` 的扩展要求
5. 当前仓库里不存在 `fvg` 指标实现，也没有 `fvg` 对应的 registry / catalog / config 接口。
6. `md.kline_bar` schema 已支持：
   - `1h`
   - `4h`
   - `1d`
   - `3d`
   但当前 live `orderflow` 库中，`ETHUSDT` 的 `futures/spot + 1h/4h/1d` 行数为 `0`。
7. 当前 `kline_history` 只显式输出：
   - `1m`
   - `15m`
   - `1h`
   - `1d`
   没有 `4h`，且它本身是上下文包，不是 `fvg` 的正式结构输出。
8. 因此 `fvg` 不能只依赖现有 `kline_history` snapshot 做二次拼装；必须新增正式指标实现，并保证其 HTF OHLC 数据在数据库中可核对。

### 2.2 当前事件持久化现状

`Dispatcher` 当前处于 storage-minimal 模式，只写 `feat.indicator_snapshot`，并跳过 `evt.*` 等事件表持久化。

涉及文件：

1. [`/data/systems/indicator_engine/src/runtime/dispatcher.rs`](/data/systems/indicator_engine/src/runtime/dispatcher.rs)

结论：

- 当前数据库里的 `evt.indicator_event` / `evt.divergence_event` 不是完整、持续可信的数据源。
- 7 天事件明细不能依赖事件表回拼。
- 原始层 7 天事件明细必须在 `indicator_engine` 内部基于当前滚动历史重跑检测函数得到。

### 2.3 当前数据库结构

已确认的真实表结构：

1. `feat.indicator_snapshot.payload_json` 为 `jsonb`，可直接承载新增 `recent_7d` 结构。
2. `evt.indicator_event` 当前没有 `event_id` 列。
3. `evt.divergence_event` 当前没有 `event_id` 列。
4. 当前不存在专用 typed 事件表：
   - `evt.absorption_event`
   - `evt.initiation_event`
   - `evt.exhaustion_event`
5. `feat.indicator_snapshot` 已有幂等唯一键：
   - `(ts_snapshot, symbol, indicator_code, window_code)`

涉及库表：

1. `feat.indicator_snapshot`
2. `evt.indicator_event`
3. `evt.divergence_event`
4. `evt.absorption_event`（待新增）
5. `evt.initiation_event`（待新增）
6. `evt.exhaustion_event`（待新增）

结论：

- `snapshot` 表本次不需要新增列。
- `evt.indicator_event` / `evt.divergence_event` 若恢复持久化，必须补 `event_id` 与唯一索引，否则 replay/backfill 后无法稳定去重。
- `absorption / initiation / exhaustion` 不能继续只依赖 generic 事件表；必须新增 typed 事件表作为主审计源。

### 2.4 当前历史深度是否足够

`StateStore` 当前保留 `8d` 分钟历史，启动回放也保证至少 `7d`。

涉及文件：

1. [`/data/systems/indicator_engine/src/runtime/state_store.rs`](/data/systems/indicator_engine/src/runtime/state_store.rs)
2. [`/data/systems/indicator_engine/src/app/runtime.rs`](/data/systems/indicator_engine/src/app/runtime.rs)

结论：

- 现有 `8d` 保留窗口只能视为“输出最近 `7d` 事件明细”的最低基线，不应表述为已经充分满足。
- 本次不应降低该窗口。
- 对 `absorption` 这类额外依赖长 lookback/warmup 的检测器，`8d` 几乎没有容错；一旦内存或 canonical 历史存在缺口，最老一段 `recent_7d` 就会退化为部分覆盖。
- 更稳妥的目标是保留 `9d+`，或者至少把 canonical DB 历史补足能力视为 completeness 前置条件，而不是可选优化。
- 若未来事件检测因 lookback 再扩展，优先增加 `HISTORY_LIMIT_MINUTES`，不要偷裁 `recent_7d`。
- 但实现上不能假设内存历史永远完整；必须设计统一的历史补足与降级策略。
- `recent_7d` 的输出窗口虽然是 `7d`，但检测输入历史不能只取刚好 `7d`；必须额外包含 detector-specific warmup buffer。

### 2.5 历史补足策略必须统一

本次必须把“历史不足时怎么办”写成统一策略，而不是由每个指标各自兜底。

统一策略：

1. 第一优先级：使用内存中的滚动历史。
2. 第二优先级：若内存历史不足，回补数据库中已落地的 canonical 历史。
3. 第三优先级：若内存与数据库都不足，则输出“当前可见历史”下的部分结果。
4. 任何部分结果都必须显式带出覆盖度字段，禁止把不完整历史伪装成完整 7 天结果。

适用范围：

1. 本次 5 个主事件指标的 `recent_7d`
2. 后续所有依赖滚动历史的指标
3. 尤其是 `15m / 4h / 1d / 7d` 这类窗口型指标

结论：

- “有多少输出多少”可以接受。
- 但前提是必须显式告诉上层：这次只覆盖了多少历史、缺了多少。
- 对 `llm` 而言，带覆盖度的部分结果优于静默缺失或伪完整结果。
- 对 `fvg` 这类 HTF 结构指标，同样适用该原则，只是覆盖度单位应优先表达为：
  - `requested_bars`
  - `returned_bars`
  - `missing_bars`
  而不是只用分钟数。

## 3. 不妥协的需求边界

本次实现必须同时满足：

1. 保留现有当前分钟字段语义。
2. 新增最近 `7d` 全量事件明细。
3. `recent_7d.events` 统一按 `event_available_ts` 升序，也就是 `旧 -> 新`。
4. `recent_7d.events` 只收“已确认并可交易”的事件，不收未确认 candidate。
5. 每条事件必须有稳定 `event_id`，可跨 replay/backfill 去重。
6. 原始层优先保留 raw 字段；normalized/confirm score 可并存，但不能替代 raw。
7. `temp_indicator` 层允许变大；不能为了省存储删掉 7 天明细。
8. 如果当前事件表无法承载需求，应恢复或重建事件落库，不以“现在没落库”为理由降低需求。
9. 历史不足时必须按“内存 -> 数据库 -> 部分输出”的统一策略补足，不能直接返回空。
10. 所有部分结果必须显式附带覆盖度与来源字段，不能静默降级。
11. 扩展后的最终指标 payload 必须持续落到 `feat.indicator_snapshot`，MQ minute bundle 只能作为传输链路，不能替代数据库审计落库。
12. `absorption / initiation / exhaustion` 必须新增 typed 事件表，并以 typed 表作为主审计源。
13. `event_id` 必须在检测层或共享 helper 生成，禁止在 writer 层临时拼接。
14. typed 事件表写入默认使用 `UPSERT`，不能只做 `DO NOTHING`。
15. `_all_history()` 必须共用核心检测逻辑，不能复制一份 current 检测函数再局部改动。
16. `divergence` 如需做极端保护，不允许静默截断；任何截断都必须显式输出 `truncated` 与 `truncated_count`。
17. `price_volume_structure` 原始产出层必须保留现有 `15m / 4h / 1d`，并新增 `3d` 窗口；不接受只在 `llm` 层临时拼装 `3d`。
18. 对 `price_volume_structure / footprint / orderbook_depth / liquidation_density` 这类 level-heavy 指标，数据库中必须保留可核对的 raw level 数据；不允许只有 `levels_count` / 摘要而没有对应 raw 表。
19. `feat.indicator_snapshot` 与其配套 raw level 落库必须先于 MQ minute bundle 对外可见；若不采用“先落库后发布”，则必须引入等价 outbox 机制保证一致性。
20. replay / rewind / repair 工具必须同步清理并重建：
   - `feat.indicator_snapshot`
   - `feat.indicator_level_value`
   - `feat.liquidation_density_level`
   - `evt.*`
   不能只回卷 snapshot。
21. 除 migration 外，`sql/rebuild_orderflow.sql` 与 `sql/rebuild_orderflow_ops.sql` 也必须同步更新，保证新环境建库结果与增量迁移一致。
22. 必须新增 `fvg` 结构指标；基础版判定市场固定为 `futures`，不允许把 `spot` 数据混入基础 FVG 成立条件。
23. `fvg` 基础版必须正式输出：
   - `1h`
   - `4h`
   - `1d`
   三档窗口；若未来只保留 `4h/1d`，必须显式改配置，不允许静默删除 `1h`。
24. `fvg` 每条结构区必须有稳定 `fvg_id`，可跨分钟、replay、repair 持续追踪同一结构区。
25. `fvg` 的基础判定、`fill_pct`、`touch_count`、`state`、`age_bars` 只允许基于同一 HTF 的已收盘 bar 序列计算；不允许把未收盘 HTF bar 混入基础状态机。
26. `fvg` 的可选增强字段：
   - `overlap_lvn`
   - `inside_value_area`
   - `distance_to_avwap`
   - `spot_confirm_at_birth`
   不参与基础 FVG 是否成立的判定，只能作为下游共振打分附加信息。
27. `fvg` 的完整 active zone 数组必须持续落到 `feat.indicator_snapshot.payload_json`，不允许只保留最近一条或只保留摘要。
28. `fvg` 的原始结构区必须可在数据库直接核对；如果 snapshot 后续存在压缩风险，必须同步把 active zone 扇出到 raw 审计表，不允许只存在 MQ / 临时文件。
29. 若当前 `md.kline_bar` 未持续落 `futures 1h/4h/1d`，必须补齐 canonical HTF kline 落库；可用 `md.agg_trade_1m` 聚合回填，但不能长期停留在“只在内存临时拼 HTF bar”的状态。

### 3.1 最终实施硬规则

以下 12 条作为本次改造的最终实施硬规则，优先级高于实现细节。若具体编码方案与下列规则冲突，以本节为准。

1. `divergence` 不允许 silent cap；若因极端性能保护发生截断，必须显式输出：
   - `truncated`
   - `truncated_count`
   - `candidate_count_before_truncation`
2. typed 事件表是主审计源：
   - `evt.absorption_event`
   - `evt.initiation_event`
   - `evt.exhaustion_event`
   - `evt.divergence_event`
   generic `evt.indicator_event` 不默认双写这 3 类新 typed 事件。
3. `absorption` 相关 `stacked_*` 输出除：
   - `stacked_buy_trigger`
   - `stacked_sell_trigger`
   - `max_buy_stack_len_trigger`
   - `max_sell_stack_len_trigger`
   之外，还必须显式输出：
   - `trigger_side`
   不允许仅依赖 `direction + stacked_*` 让上层推断触发方。
4. `initiation.follow_through_*` 必须同时输出：
   - 固定门槛值：`min_follow_required_minutes`
   - 实际观测值：`follow_through_minutes`、`follow_through_delta_sum`、`follow_through_hold_ok`、`follow_through_max_adverse_excursion_ticks`
   且 `follow_through_max_adverse_excursion_ticks` 优先基于 `high/low` 计算，不以 `close` 替代。
5. typed 事件写入超时不能阻断主流程，但必须满足：
   - 使用 `timeout(3s)` 或等价超时控制
   - 失败后仅 `warn!`，不阻断 `feat.indicator_snapshot` 与 MQ 发布
   - `warn` 必须限流或聚合，禁止日志洪水
   - 必须暴露 health metrics，至少包含：`consecutive_failures`、`last_failure_ts`、`last_failure_reason`
6. migration 必须拆分为三个阶段：
   - 新表创建
   - 旧表去重
   - 旧表并发索引补齐
   对已有表新增唯一索引时，不允许跳过“先去重”步骤。
7. `_all_history()` 的性能保护不允许悄悄改变“最近 7 天全量事件明细”的语义。任何保护性裁剪、降级或部分输出，都必须通过显式字段对上层可见。
8. `feat.indicator_snapshot` 与 raw level 审计表属于 audit-durable 数据；它们未成功持久化时，不允许 minute bundle 先对外发布。typed 事件表允许按前述规则降级，但 snapshot/level 不允许静默跳过。
9. `fvg` 基础判定市场固定为 `futures`；spot 只能进入可选 confluence 字段，不能进入基础判定阈值。
10. `fvg.active_bull_fvgs` / `fvg.active_bear_fvgs` 只允许包含：
   - `fresh`
   - `partially_filled`
   - `mitigated`
   不允许把 `fully_filled` 或 `invalidated` 混进 active 数组。
11. `fvg` 在 `1h / 4h / 1d` 每个窗口下都必须先输出数组：
   - `fvgs[]`
   用这个数组列出该窗口内全部 FVG；不允许只保留单条对象或只保留 `nearest_*`。
12. `fvg.nearest_bull_fvg` / `fvg.nearest_bear_fvg` 必须从 `fvgs[]` 过滤后的 active 集合中按“价格到区间 `[lower, upper]` 的最小距离”选出，不允许偷简化成“最新出生”或“数组首项”。
13. `fvg_id` 在结构区的整个生命周期内必须稳定；状态演化只允许改变：
   - `fill_pct`
   - `touch_count`
   - `state`
   - `age_bars`
   不允许每分钟重生一条新 ID。

## 4. 共享改造项

### 4.1 事件检测函数统一为“共享核心 + current wrapper + 7d builder”

对以下 5 个主指标，统一引入 `_all_history()` 或等价变体：

1. `divergence`
2. `absorption`
3. `initiation`
4. `buying_exhaustion`
5. `selling_exhaustion`
6. `price_volume_structure`（窗口扩展：新增 `3d`）

要求：

1. `_all_history()` 返回当前滚动历史中全部已确认事件。
2. 现有当前分钟输出必须通过 `_all_history()` 结果过滤得到。
3. 不允许维护两套独立检测逻辑，以免 current 与 7d 口径漂移。
4. 推荐结构：
   - 核心检测器：返回全历史已确认事件
   - current wrapper：过滤 `current.events`
   - `recent_7d` builder：过滤最近 `7d`

统一过滤关系：

1. `current.events` = `_all_history()` 中 `event_available_ts == 当前分钟` 的子集
2. `recent_7d.events` = `_all_history()` 中 `event_available_ts >= ts_bucket - 7d` 的子集
3. `latest_7d` = `recent_7d.events` 最后一条

### 4.2 每条事件统一补齐公共字段

对 5 类主事件统一补齐：

1. `event_id`
2. `event_start_ts`
3. `event_end_ts`
4. `event_available_ts`
5. `confirm_ts`
6. `type`
7. `direction`
8. `score`

其中：

- `event_id` 是去重主键，不可省略。
- `score` 在事件 JSON 中统一命名，不继续散落为各指标内部自定义名。

### 4.3 统一 event_id 规则

建议统一规则：

```text
<indicator_code>|<symbol>|<type>|<direction>|<event_available_ts>|<key_anchor>
```

其中 `key_anchor` 按指标选：

1. `divergence`: `pivot_ts_1 + pivot_ts_2 + pivot_side`
2. `absorption`: `event_start_ts + event_end_ts + price_low + price_high`
3. `initiation`: `event_start_ts + event_end_ts + pivot_price`
4. `buying_exhaustion`: `pivot_ts_1 + pivot_ts_2 + type`
5. `selling_exhaustion`: `pivot_ts_1 + pivot_ts_2 + type`

要求：

1. 同一事件 replay/backfill 后 `event_id` 必须稳定。
2. 不允许用随机 UUID。
3. 不允许用仅依赖数组下标的瞬时 ID。
4. `event_id` 必须在检测层或共享 helper 生成，并同时进入：
   - typed event row
   - snapshot payload
   - DB writer
5. 不允许在 writer 阶段再重新推导 `event_id`。

### 4.3A 统一 fvg_id 规则

`fvg` 虽然不是事件型指标，但同样需要稳定主键，便于：

1. 跨分钟追踪同一结构区状态
2. replay / repair 后对齐同一条结构区
3. snapshot 与 raw 审计表交叉核对

建议规则：

```text
fvg|<symbol>|<tf>|<side>|<birth_ts>|<lower>|<upper>
```

要求：

1. 同一结构区在 replay/backfill 后 `fvg_id` 必须稳定。
2. 不允许把 `fill_pct`、`state`、`touch_count`、`age_bars` 这类会随时间变化的字段拼进 `fvg_id`。
3. `fvg_id` 必须同时进入：
   - snapshot payload
   - raw 审计表
4. 不允许在 writer 阶段再重新推导 `fvg_id`。

### 4.4 快照 payload 不做事件压缩

`snapshot_writer` 当前只压缩：

1. `price_volume_structure`
2. `footprint`
3. `orderbook_depth`
4. `kline_history`

本次明确要求：

- 不要新增对这 5 个主事件指标的摘要压缩。
- `payload_json` 必须完整保留 `recent_7d.events`。
- 对当前已经做 payload 压缩的 level-heavy 指标：
  - `price_volume_structure`
  - `footprint`
  - `orderbook_depth`
  - `kline_history`

  - `fvgs`
  - `active_bull_fvgs`
  - `active_bear_fvgs`
  - `nearest_bull_fvg`
  - `nearest_bear_fvg`
  做摘要压缩；这些结构区字段必须原样入库到 `feat.indicator_snapshot.payload_json`。

涉及文件：

### 4.5 恢复事件落库作为审计与回放辅助

虽然 `recent_7d` 的直接数据源不依赖 DB，但本次建议同步恢复事件落库，原因：

1. 便于人工排查与 SQL 复核。
2. 便于后续离线聚合、事件仓分析。
3. 便于验证 `recent_7d.events` 与事件表是否一致。

要求：

1. 重新启用 `Dispatcher` 中对事件表的写入。
2. 新增 typed 事件表：
   - `evt.absorption_event`
   - `evt.initiation_event`
   - `evt.exhaustion_event`
3. 为 `evt.indicator_event` / `evt.divergence_event` 增加 `event_id`。
4. 为事件表增加唯一索引，避免 replay/backfill 重复。

注意：

- typed 事件表是主审计源；generic `evt.indicator_event` 只保留为兼容层或后续统一视图输入，不再作为三类新事件的唯一落库目标。
- 事件表恢复持久化是“辅助审计能力”，不是 `recent_7d` 的主来源。
- 即使事件表暂时为空，也不影响 `recent_7d` 的构建逻辑。

### 4.6 统一历史补足与覆盖度输出

本次除事件指标外，也应把该策略定义成后续所有滚动历史指标的统一基线。

统一补足顺序：

1. 先取内存滚动历史
2. 缺口部分再取数据库 canonical 历史
3. 若仍不足，则按当前可见历史继续计算
4. 对任何 `_all_history()` 检测器，输入历史都必须是：
   - `7d` 输出窗口
   - 加上 detector-specific warmup buffer

统一覆盖度字段要求：

1. `history_requested_minutes`
2. `history_covered_minutes`
3. `history_missing_minutes`
4. `history_coverage_ratio`
5. `history_source`

其中 `history_source` 建议枚举：

1. `memory_only`
2. `memory_plus_db`
3. `partial_memory_only`
4. `partial_memory_plus_db`

### 4.6A FVG 的统一输出与状态更新原则

`fvg` 不走“事件数组 + recent_7d”这条路径，而是走“HTF 结构区 + 生命周期推导”。

统一原则：

1. `fvg` 的基础扫描对象是同一 TF 的已收盘 bar 序列，不是分钟事件序列。
2. 基础输出窗口固定为：
   - `1h`
   - `4h`
   - `1d`
3. 每个窗口都必须输出：
   - `fvgs`
   - `active_bull_fvgs`
   - `active_bear_fvgs`
   - `nearest_bull_fvg`
   - `nearest_bear_fvg`
4. `fill_pct / touch_count / state / age_bars` 统一由共享无状态推导逻辑维护：每分钟基于当前可得的已收盘 HTF bar 序列重扫并推导，不允许为 `fvg` 在 `StateStore` 中引入跨分钟持久状态机。
5. `fvgs[]` 是每个窗口的主数组输出，必须列出该窗口当前全部可参考 FVG，并用 `side` 区分 bull/bear。
6. `active_*_fvgs` 只保留：
   - `fresh`
   - `partially_filled`
   - `mitigated`
   两类失活状态：
   - `fully_filled`
   - `invalidated`
   不进入 active 数组，但可保留在原始审计明细中。
7. 覆盖度统一输出：
   - `requested_bars`
   - `returned_bars`
   - `missing_bars`
   - `coverage_ratio`
   - `history_source`
8. `nearest_*` 必须在 `fvgs[]` 过滤后的 active 集合上计算，不允许从历史所有 FVG 中选。

### 4.6B 持仓后可见性与 current-only 指标

当前还有一类风险，不属于“历史补足失败”，而是“指标本身只暴露当前 1m 状态”，会让开仓时的关键微结构条件在持仓后直接消失。

本次明确区分三类：

1. 已在本轮改造中覆盖的 current-only 事件指标：
   - `divergence`
   - `absorption`
   - `initiation`
   - `buying_exhaustion`
   - `selling_exhaustion`
   - 以及其 directional filtered views
   这些通过 `recent_7d / latest_7d` 解决“开仓后看不到历史关键事件”的问题。
2. 必须在本轮一起优化的非事件型 current-only 指标：
   - `orderbook_depth`
   - `liquidation_density`
   这两项直接影响持仓后模型是否还能看到开仓时的关键微结构/爆仓区背景，不能延期到下轮。

3. 已有部分窗口化输出、但仍保留 current `1m` sidecar 的次一级风险指标：
   - `high_volume_pulse`
   - `ema_trend_regime`
   - `price_volume_structure`
   这些不属于“纯 current-only”，因为已经存在窗口化或前向填充视图；本轮先补语义约束，不强制全部扩成完整历史数组。

结论：

1. `orderbook_depth` 当前实现只有 `window_code="1m"` 的 snapshot；`obi / ofi / microprice / fake_order_risk / exec_confirm / spot_confirm / levels` 都会随分钟滚动覆盖，持仓后看不到开仓时微结构条件。
2. `liquidation_density` 当前实现也只有 `window_code="1m"` 的 snapshot；开仓前后的关键爆仓峰值区会直接被新一分钟覆盖。
3. 这两个指标不能继续只给“当前值 + raw DB 审计表”；还必须在原始层输出可供持仓后模型继续使用的窗口化记忆视图。
4. `high_volume_pulse.intrabar_poc_*`、`ema_trend_regime.ema_13 / ema_21 / ema_34 / ema_band_*`、`price_volume_structure` 顶层 `1m` snapshot 仍然是 current sidecar，但因为已经有 `by_window` / `trend_regime_by_tf` / `ffill_series_by_output_window` 等窗口视图，本轮先要求文档与实现显式标注“current sidecar”语义，不把它们误当成持仓记忆层。
5. 事件型指标 `divergence / absorption / initiation / buying_exhaustion / selling_exhaustion` 不再通过“补一个 1m 窗口”解决，而是统一通过 `recent_7d / latest_7d` 解决持仓后可见性问题。

本次最低实施要求：

1. `orderbook_depth` 必须新增持仓后可见的 `by_window` 或等价记忆视图，至少覆盖：
   - `15m`
   - `1h`
   用于保留最近窗口内的 `obi / ofi / microprice / confirm / fake_order_risk` 等关键信号。
2. `liquidation_density` 必须新增窗口化输出，至少覆盖：
   - `15m`
   - `1h`
   - `4h`
   - `1d`
   - `3d`
   使关键爆仓峰值区在持仓后仍能被模型看到，而不是只剩当前 1m 摘要。
3. `high_volume_pulse`、`ema_trend_regime`、`price_volume_structure` 本轮至少要在文档与实现上明确：
   - 顶层 `1m` 字段只是 current sidecar
   - 持仓后模型应优先读取其窗口化输出
   - 不得把顶层 `1m` current 值误当作历史记忆层

事件型指标至少在 `recent_7d` 层补：

1. `lookback_requested_minutes = 10080`
2. `lookback_covered_minutes`
3. `lookback_missing_minutes`
4. `lookback_coverage_ratio`
5. `history_source`

要求：

1. 若数据库也缺历史，`recent_7d.events` 仍应输出当前可见的事件。
2. 但 `latest_7d` 与 `event_count` 的解读必须受覆盖度约束。
3. 上层聚合与 `llm` 提示词后续可以据此判断“7d 偏置”是否可信。

### 4.8 统一 warmup buffer 原则

`recent_7d` 的输出窗口固定为最近 `7d`，但各检测器的输入历史必须额外带 warmup buffer。

要求：

1. `divergence`
   - 需要覆盖 pivot leg gap、detrend window、robust z lookback
2. `initiation`
   - 需要覆盖 `ZDELTA_LOOKBACK`、`MIN_FOLLOW_MINUTES`、`MAX_EVENT_MINUTES`
3. `absorption`
   - 需要覆盖事件最长持续分钟数与确认窗口
4. `buying_exhaustion / selling_exhaustion`
   - 需要覆盖 pivot leg gap 与 confirm bars

原则：

1. 输入历史长度不允许只取 `10080` 分钟整。
2. 输出时再按 `event_available_ts >= ts_bucket - 7d` 裁切。
3. warmup buffer 不单独暴露给上层，但覆盖度必须按实际输入后再折算到 7 天窗口。

### 4.7 统一历史补足策略的逐文件实现项

3. 对所有滚动窗口指标输出统一覆盖度元信息。

必须改造：

职责：

## 15. 最终判断

从代码与数据库现状来看，本次需求如果要“不妥协实现”，必须接受以下事实：

1. 7 天事件明细的主来源必须是 `indicator_engine` 内部重跑检测函数，不是 DB。
2. 5 个主事件指标都需要新增 `_all_history()` 路径，不能只在现有 current 逻辑上打补丁。
3. 事件表必须补 `event_id` 并恢复落库，否则后续审计、对账、重放都会非常脆弱。
4. `snapshot.payload_json` 会明显变大，但这是需求优先下必须接受的成本。
5. 实现不能假设历史永远完整，必须统一支持“内存 -> 数据库 -> 部分输出”的历史补足策略。
6. 对 level-heavy 指标，必须把 raw level 审计表一起恢复；不能接受只有 snapshot 摘要、没有价位明细的落库方案。
7. 发布链路必须满足“snapshot/raw level 先成功持久化，再对外发布 minute bundle”或等价 outbox，一致性不能靠运气。
8. 回放、修复、建库 SQL 需要一起升级；只改 runtime 不改工具链，后续审计一定失真。
9. `fvg` 需要稳定、可核对的 HTF OHLC 输入；当前 live 库 `md.kline_bar` 为空这一点不能被忽略，必须补足 canonical HTF kline 链路。
10. `fvg` 虽然不是事件指标，但它同样必须有稳定主键、snapshot 落库和 raw 审计落库，否则后续很难核对同一结构区的状态演化。

达到以上 10 点后，这套设计才足以支撑后续 `llm` 层做：

1. 最近 `30m / 4h / 1d / 7d` 事件统计
2. 连续 absorption / initiation / exhaustion 序列识别
3. 事件方向偏置分析
4. 价格是否处于 HTF FVG 的回补 / 失效 / 再定价区
5. 更高质量的中大型交易机会判断
