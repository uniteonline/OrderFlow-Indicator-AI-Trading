## 1. 评审结论（是否符合你的要求）

**结论：整体设计方向正确，基本符合你的要求，可以作为实现蓝图。**

你当前设计已经满足这几个关键点：

1. **两个独立子系统**（`Market Data Ingestor` / `Indicator Engine`）✅  
2. **系统间通过 RabbitMQ 通信**，指标系统不直连交易所 ✅  
3. **按 `指标.md #2` 采集，按 `指标.md #1` 计算** 的职责分离 ✅  
4. **指标一文件**，且共享逻辑集中管理，降低改动串扰 ✅  
5. **落库对齐 TimescaleDB 表结构**（`md.* / feat.* / evt.* / ops.*`）✅

但为了后续编码更稳、避免返工，以下约束在本设计中**固定为必选项**（都不改变你的大方向）：

### 必补的工程约束（已纳入本文）

1. **MQ 消息契约版本化（schema_version）**  
   否则后续你改字段会把指标引擎打挂。

2. **Ingestor “先写库再发MQ”的失败策略明确化**  
   需要定义：写库成功但发 MQ 失败 / 发 MQ 成功但进程崩溃 的处理规则（幂等重发）。

3. **Indicator Engine 的事件时间（水位线/watermark）机制**  
   depth、trade、bbo 到达不同时序，如果没有 watermark，3/6/7/12/13 等事件类指标会抖动严重。

4. **指标输出与落库写入解耦**  
   指标计算线程不直接同步写 DB，建议统一交给 `storage writer workers` 批量落库。

5. **3d Kline 明确为本地聚合产物（derived）**  
   不走交易所原生流；要在目录里明确放在 `aggregate/` 或 `state/` 模块，避免后续误当 WS 数据。

6. **高频原始表幂等写入必须有数据库唯一约束**  
   特别是 `md.trade_event` / `md.depth_delta_l2`，否则重连和补洞时会重复写入并污染指标结果。

7. **`mark_price` 与 `funding_rate` 路由语义分离**  
   不再使用模糊的 `mark_funding` 数据集名，避免消费端歧义。

8. **文档单一事实源（SSOT）**  
   `指标.md` 仅保留根目录一份，`docs/` 只做引用，不做镜像副本。

## 2. 总体架构（你这套方案的正确形态）

```text
Binance WS/REST
   │
   ▼
[Market Data Ingestor]
   ├─ 标准化（trade/depth/bbo/kline/mark-funding/force-order）
   ├─ Gap检测 + REST补洞 + 3d本地聚合
   ├─ 写 TimescaleDB: md.*, ops.*
   └─ 发布 RabbitMQ: x.md.live (标准化实时消息)

可选 [MD Replayer]
   ├─ 读取 TimescaleDB: md.*（历史区间）
   └─ 发布 RabbitMQ: x.md.replay（标准化回放消息）

RabbitMQ (x.md.live / x.md.replay / x.ind / x.dlx)
   │
   ▼
[Indicator Engine]
   ├─ 按 consume_mode 消费 x.md.live 或 x.md.replay
   ├─ 18个指标独立文件计算（共享数学模块复用）
   ├─ 写 TimescaleDB: feat.*, evt.*, ops.*
   └─ 发布 x.ind (指标结果/事件，供告警/策略/监控消费)
```

> 这是符合你“采集和指标拆成 2 个系统”的最佳形态：**数据面用 MQ，控制面可后续补 API**（当前阶段可以不做 API）。

## 5. 两个子系统职责边界（必须严格执行）

## 5.1 Market Data Ingestor（只做采集与标准化）

### 职责

1. 连接 Binance WS/REST（按 `指标.md #2` 的数据源清单）
2. 标准化原始消息字段（spot / futures 字段映射统一）
3. 检测 gap、调度 REST 补洞
4. 执行 depth 快照一致性状态机（`INIT -> SNAPSHOT_SYNCING -> LIVE_SYNCED -> DESYNC`）
5. 1d -> 3d 本地聚合（如果你采用多周期直存且 3d 本地生成）
6. 写入 `md.*` 和 `ops.*`
7. 发布标准化消息到 `x.md.live`

### depth 快照一致性状态机（固定）

1. `INIT`：初始态，不处理 diff 计算。  
2. `SNAPSHOT_SYNCING`：拉取快照并对齐 `U/u/pu` 链。  
3. `LIVE_SYNCED`：链路连续，允许实时增量更新。  
4. `DESYNC`：链路断裂或校验失败，立即触发补洞/重建并回到 `SNAPSHOT_SYNCING`。

### 不做的事（非常重要）

- 不做任何指标判定（不算 OBI/OFI/CVD/VPIN 等）
- 不发布策略信号
- 不把业务判定掺进标准化层

> 这样才能保证 Ingestor 是可复用的数据底座。

---

## 5.2 Indicator Engine（只做指标计算与事件输出）

### 职责

1. 按 `consume_mode` 消费标准化行情：
   - `live`：消费 `x.md.live`
   - `replay`：消费 `x.md.replay`
2. 构建窗口与运行态缓存（100ms / 1s / 1m / 15m / 1h / 4h / 1d / 3d）
3. 执行 18 个指标（每指标独立文件）
4. 写入 `feat.*` / `evt.*`
5. 发布指标结果与事件到 `x.ind`
6. 重启 warm-up 时，可从 `md.*` 只读回放最近窗口重建运行态（不反向修改 `md.*`）

### 可选能力（建议保留扩展位）

- backfill 模式重算（通过消息头 `backfill_in_progress` 控制是否抑制事件发射）

### 状态缓存演进约束（state_store）

当前阶段保留 `state_store.rs` 单入口；当状态复杂度上升时，按下面方向拆分：

1. `market_state/`：trade/bbo/depth/kline 等市场运行态。  
2. `indicator_state/`：每个指标自身缓存与中间态。  

> 该演进是目录约束，不要求当前阶段立即拆分。

## 9. 你当前设计里建议补充的“运行时约束”

这些不加也能写代码，但加了会大幅降低实盘联调难度。

### 9.1 启动顺序

1. RabbitMQ
2. PostgreSQL + TimescaleDB
3. `market_data_ingestor`
4. `indicator_engine`

### 9.2 发布顺序（Ingestor）

**固定方案：Transactional Outbox（禁止直接“写库后立即发MQ”）**。  
采用以下顺序，才能消除“写库成功但消息丢失”的窗口：

1. 同一数据库事务内：
   - 写 `md.*`（标准化事件）
   - 写 `ops.outbox_event`（待投递消息，含 `message_id/schema_version/routing_key/payload`）
2. 事务提交后，由 `outbox_dispatcher` 异步扫描 `pending` 记录（`FOR UPDATE SKIP LOCKED`）。
3. 发布到 `x.md.live`（persistent + publisher confirm）。
4. 收到 confirm 后，将 outbox 状态改为 `sent`；失败则增加 `retry_count` 并指数退避重试。
5. 进程重启时先恢复未发送 outbox，再进入实时流。

> 这样可以保证：只要 `md.*` 入库成功，消息最终可达（at-least-once）。

### 9.3 幂等键（至少约定）

1. 生产侧（Ingestor）幂等键：
   - trade：`market + symbol + agg_trade_id`
   - depth：`market + symbol + final_update_id`
   - kline：`market + symbol + interval + open_time`
2. 消费侧（Indicator）幂等键：
   - feature/event：`symbol + indicator_code + window + ts + calc_version`
3. 存储层必须落实：
   - 高频原始表唯一索引（见第 7 章 SQL 补丁）
   - 写入采用 `INSERT ... ON CONFLICT DO NOTHING/UPDATE`
4. MQ 幂等：
   - 以 `message_id` 做消费去重缓存（短 TTL）+ 业务键二次去重。

### 9.4 backfill 模式

REST 补洞回放时，Indicator Engine 建议：
- 可计算 feature 并写库
- 对事件类指标（3/6/7/12/13）**默认降级告警/不外发策略信号**（避免误报）

### 9.5 文档单一事实源（SSOT）

1. 指标定义唯一来源：根目录 `指标.md`。  
2. `docs/指标索引.md` 只做跳转和版本说明，不复制正文。  
3. 若发生冲突，始终以根目录 `指标.md` 为准。  

### 9.6 Storage Writer 批量 Flush 策略（模板，后续可调）

`indicator_engine/src/storage/batch_flush.rs` 建议固定支持以下触发条件：

1. 按条数触发：累计 `N` 条即 flush。  
2. 按时间触发：达到 `T ms` 即 flush。  
3. 按窗口事件触发：收到窗口关闭事件（如 `1m/15m` close）即 flush。  
4. 程序退出触发：`drain + flush`，确保缓冲区不丢。  

> 该策略用于解耦“指标计算线程”与“落库线程”，并降低高频逐条写入开销。

### 9.7 `ops.service_heartbeat` payload 统一约定

建议 heartbeat `metrics_json` 至少包含：

1. `role`  
2. `instance_id`  
3. `last_processed_event_ts`  
4. `event_lag_ms`  
5. `queue_lag`（可获取时上报）  
6. `error_rate`  
7. `backfill_mode`  

> Ingestor 与 Indicator Engine 统一该字段集合，可显著提升排障效率与运行可观测性。

### 9.8 冷启动（完全 MQ 驱动）可执行步骤

目标：Indicator Engine 在冷启动时不依赖 `md.*` 历史查询，完全通过 `x.md.live` 消息恢复运行态（包含 `md.orderbook_snapshot_l2`）。

#### 前置条件（必须）

1. 已声明并生效队列绑定：`q.indicator.md.orderbook` 订阅  
   - `md.*.depth.ethusdt`  
   - `md.*.orderbook_snapshot_l2.ethusdt`  
   - `md.*.bbo.ethusdt`  
2. Ingestor 实现 `md.orderbook_snapshot_l2` 发布（`source_kind=derived`）。  
3. Indicator 实现“先等快照、后放行 depth 增量”的启动门控。  

#### 执行步骤（操作序列）

1. 停止两个业务进程，仅保留基础设施：RabbitMQ + PostgreSQL/TimescaleDB。  
2. 启动 `market_data_ingestor`，等待拓扑幂等声明完成（exchange/queue/binding 可重复声明）。  
3. Ingestor 进入 `SNAPSHOT_SYNCING`：  
   - 先订阅 depth 增量并缓存；  
   - 拉取 REST depth 快照；  
   - 生成并发布 `md.orderbook_snapshot_l2`（`routing_key=md.{market}.orderbook_snapshot_l2.{symbol}`）；  
   - 再回放缓存 depth，切到 `LIVE_SYNCED`。  
4. 启动 `indicator_engine`，消费 `x.md.live`：  
   - 收到 `md.orderbook_snapshot_l2` 前，只缓存 orderbook 相关消息，不执行依赖盘口状态的指标计算；  
   - 收到快照后初始化本地 orderbook state；  
   - 仅应用 `final_update_id > snapshot.last_update_id` 的 depth 增量；  
   - 增量链连续后，将 `orderbook_ready=true`。  
5. 当以下条件同时满足时，放开事件输出：  
   - `orderbook_ready=true`；  
   - 必需窗口（如 `1m`）已完成 warm-up；  
   - `event_lag_ms` 低于阈值；  
   - `backfill_in_progress=false`。  
6. 冷启动完成后进入常规运行：实时消费 + 周期 flush + heartbeat 上报。  

#### 验收检查（可直接执行）

1. 快照是否已入库：  
```sql
SELECT market, symbol, ts_snapshot, depth_levels, last_update_id
FROM md.orderbook_snapshot_l2
WHERE symbol = 'ETHUSDT'
ORDER BY ts_snapshot DESC
LIMIT 5;
```

2. 快照后是否持续有 depth 增量：  
```sql
SELECT market, symbol, ts_event, final_update_id
FROM md.depth_delta_l2
WHERE symbol = 'ETHUSDT'
ORDER BY ts_event DESC
LIMIT 20;
```

3. 两个进程心跳是否健康：  
```sql
SELECT service_name, instance_id, heartbeat_at, status, metrics_json
FROM ops.service_heartbeat
ORDER BY heartbeat_at DESC
LIMIT 20;
```

#### 失败回退（固定动作）

1. 若 Indicator 启动后 `X` 秒未收到 `md.orderbook_snapshot_l2`：重触发 Ingestor 快照重建（`DESYNC -> SNAPSHOT_SYNCING`）。  
2. 若 depth 链断裂：丢弃本地簿，重新等待下一条 `md.orderbook_snapshot_l2`，期间保持 `backfill_in_progress=true`。  
3. 回退期间允许写 feature，但事件类指标默认不外发策略信号。
